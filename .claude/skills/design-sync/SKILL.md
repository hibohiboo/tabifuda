# Design Sync

設計文書と実装の乖離を検出し、報告する手順。
原則は CLAUDE.md 最重要ルール1: **設計文書が正、実装が従。**

## 1. チェック範囲を決める

- サイクル終了時の通常チェック: `git diff <base>...HEAD` で今回触れた
  型・関数・モジュールを列挙し、それに対応する文書箇所だけ照合する
- 全量チェック(明示的に依頼された場合のみ): コアの公開API全体を対象にする

## 2. 照合する

1. 実装側: crates/ のコアから公開型を列挙する
   (Grep: `pub (enum|struct|fn|type)`)
2. docs/design/domain-model.md の対応セクションと突き合わせる。観点:
   - 型名・バリアント名・フィールド名と型が文書と一致するか
   - Event / Command / PatchOp / RuleError のバリアント一覧の過不足
   - SessionStatus の遷移規則、Actor の権限規則が文書の表と一致するか
   - serde表現の規約(タグ付き・`#[non_exhaustive]`)が守られているか
3. テストに現れる仕様(受理/拒否の期待値)も文書と突き合わせる。
   「テストが仕様」なので、テストだけ直して文書を放置しない(逆も)

## 3. 乖離の判定基準

| 状況 | 扱い |
|---|---|
| 実装が文書に無い振る舞いを持つ | 原則実装側の誤り。ただし実装側が正しい設計判断なら、文書更新案を提示して人間の承認を得てから文書を直す |
| 文書にあるが未実装 | 現フェーズのスコープ内かをタスク文書・future-requirements.md で確認。スコープ外なら乖離ではない(先回り実装しない) |
| 名前・表現だけの不一致 | 乖離として報告(用語の揺れは CLAUDE.md「用語」違反) |

## 4. 報告と修正

- 乖離一覧を表で報告する: 項目 / 文書の記述(該当行) / 実装(該当ファイル:行) / 対処案
- 乖離ゼロならその旨を1行で報告する
- 修正は**同じPRで**文書と実装の両方に入れる(別PRに分けない)
