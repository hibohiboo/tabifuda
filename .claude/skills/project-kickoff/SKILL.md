# Project Kickoff

プロジェクト構想を聞き取り、実装に入れる基盤文書一式を作るまでの手順書。
一度に全部作らない。**1文書ずつ作り、ユーザーの確認を得てから次へ進む。**
ユーザーとの対話が本体であり、文書はその記録である。

## 全体の流れ

```
0. 聞き取り(構想の受領と分岐点の特定)
1. フェーズ計画(着手順の合意)
2. ドメインモデル叩き台 → レビュー → 改訂
3. 将来要望メモ(スコープ外の安全な退避)
4. エージェント運用計画(モデル配分と改良ループ)
5. CLAUDE.md(ハーネスの中核)
6. テスト戦略(リスクベース)
7. 横断的関心事(ログ・セキュリティ・削除方針)
8. ADRで手法を明文化
```

順序は 2〜8 なら入れ替え可。ユーザーが特定文書を求めたらそこから始めてよい。
ただし 0 と 1 は飛ばさず、**7 と 8 を省略しない**(初版のこのスキルで
抜けており、ユーザー指摘で追加された経緯がある。忘れやすい)。

## Step 0: 聞き取り

ユーザーの構想を読み、**アーキテクチャを左右する分岐点**を特定して質問する。
質問は最大3つ。選択肢形式で(入力ツールがあれば使う)。定番の分岐点:

- 同期性: リアルタイム / 非同期 / ソロ(通信・インフラ設計に直撃)
- 最初のMVP: どの縦切りを最初に動かすか
- AI・外部サービスの役割: 製品内で使うか、開発でのみ使うか
- マルチテナント性、認証の要否、オフライン要否(該当しそうなら)

構想に既に答えがあるものは聞かない。技術スタックの希望は尊重し、
明白な不整合(例: 選んだインフラと同期方式の相性)だけ指摘する。

## Step 1: フェーズ計画

回答を踏まえ、フェーズ分けを提示する。原則:

1. **Phase 0 は開発基盤**(リポジトリ骨格・CI・エージェントハーネス)。
   「サイクルを回し続ける」系の目的があるなら必ず最初
2. **コアロジックを最も安い実行環境で先に検証する**
   (例: CLI版→Web版→バックエンド→デプロイ)。UIとインフラは後回し
3. 各フェーズの成果物と、後続フェーズに効く設計判断を1〜2個添える
4. 未決の分岐が計画を変える場合、その旨を明示して質問を続ける

## Step 2: ドメインモデル叩き台

references/domain-model-guide.md を読んでから作成する。要点:

- コアを純粋な状態機械として設計できないか最初に検討する。
  「操作の記録・履歴・リプレイ・監査」が要件にあればイベントソーシングを提案
- Rust/型のある言語なら主要型のスケッチをコードブロックで示す
- **末尾に「未決事項(相談したい)」を必ず置く。** 決めの問題を勝手に決めない。
  各未決事項に推奨案と理由を添える
- ユーザーの回答を受けて v0.1 に改訂し、冒頭に変更履歴を書く

## Step 3: 将来要望メモ

ユーザーが「今はやらないが将来やりたい」を出してきたら、別文書に退避する。
references/templates.md の様式に従い、各要望に必ず2つ添える:

1. **設計方針(案)**: 将来どう実現するかの1案
2. **現行モデルへの影響**: なし/小/中/大 と、その根拠

最後に「**現行版への逆輸入候補**」節を置く。実装コストほぼゼロで将来の道を
塞がない予約(空フィールドの追加、コピー方式の明記、enum拡張前提の
シリアライズ等)を抽出し、ユーザーに反映するか確認する。ここがこの文書の価値。

## Step 4: エージェント運用計画

references/agent-ops-guide.md を読んでから作成する。要点:

- 最新のモデルラインナップと価格を必ずWeb検索で確認する(記憶で書かない)
- 3層ラダー(既定の主力 / 定型用の安価モデル / エスカレーション用上位モデル)
- **エスカレーションの発動条件を明文化**(例: 2回失敗、型設計レビュー、
  フェーズ移行時)。「気分で上位モデルを使わない」を仕組みにする
- 上流フェーズほど上位モデルの価値が高い、を対応表で示す
- ハーネス改良ループ: 失敗ジャーナル(1行/件)→週次棚卸し→
  CLAUDE.md修正/スキル新設/タスク分割変更のいずれかとしてPR化

## Step 5: CLAUDE.md

references/templates.md の骨格に従う。守るべき原則:

- **最重要ルールを最大4つ、先頭に。** 長いCLAUDE.mdは守られない
- 詳細は docs/ への参照で済ませ、本体は安定した内容だけにする
  (プロンプトキャッシュ効率のため。揮発する情報を入れない)
- **用語表**を入れる。用語の揺れはエージェント誤解の最大要因
- **「やらないこと」**を入れる。特に、将来要望メモを読んだエージェントが
  未承認機能を先回り実装するのを明示的に禁止する
- **「作業の終わり方」**に、テスト実行・文書乖離チェック・
  失敗ジャーナルへの自己申告を含める

## Step 6: テスト戦略

references/test-strategy-guide.md を読んでから作成する。要点:

- **リスク表から始める**(リスク×深刻度×テストする層×手法)
- 大原則: ロジックの置き場所にテストの質量を置く。外側の層で再テストしない
- 純粋コアがあるならプロパティテスト(不変条件列挙)とゴールデン/
  リプレイテストを提案。fixture は複数目的(回帰・互換・スモーク)で使い回す
- モックしてはいけない箇所を明示(例: DBの同時実行制御はDB実物で)
- フェーズ別の導入順とCIゲートで締める

## Step 7: 横断的関心事(cross-cutting.md)

references/cross-cutting-guide.md を読んでから作成する。最低限の検討項目:

- **権限**: 認可をどの層で検証するか。ロジックをコアに寄せた設計なら
  権限もコアで(=ドメインモデルへの逆輸入が発生しうる)
- **ログの2分類**: 製品価値としてのログ(履歴・監査)と運用ログを区別し、
  運用ログにユーザー本文・個人情報を書かない規律を先に決める
- **自由入力(UGC)**: 描画時エスケープ、生HTML禁止、長さ上限。
  「データはコードではない」(スクリプト実行機構を持たない)を
  設計上の特性として明文化する
- **削除要求と履歴の折り合い**: 追記のみ・不変の原則がある設計は、
  個人情報削除との緊張を必ず持つ。「構造は不変、本文は削除可能」等の
  精密化を初期に決める
- 秘密情報の置き場、依存監査・シークレットスキャンのCI組込時期
- 最後に「フェーズ対応表」(どの項目がどのフェーズで効くか)

ドメインモデルへ影響する項目(権限など)が見つかったら、
逆輸入としてドメインモデルを改訂する(将来要望メモと同じ流儀)。

## Step 8: ADRで手法を明文化

採用した手法のブレンドを ADR 0001 として docs/adr/ に残す。
「フルXXか?」と問われたとき、部分採用の範囲と未実施部分
(例: DDD戦術のみ採用、戦略的設計は未実施でP4で再評価)を
正直に書く。違反となる変更パターン(コアへの副作用持込等)も列挙し、
レビューの根拠として使えるようにする。



- 各文書はファイルとして作成し、ユーザーに提示する。ファイル名は
  domain-model-vX.md / future-requirements.md / agent-operations.md /
  CLAUDE.md / test-strategy.md / cross-cutting.md /
  adr/0001-methodology.md に揃える(相互参照するため)
- 文書間の参照を残す(「将来要望メモ§1参照」等)。後からエージェントが
  経緯を追えることが、ハーネスとしての価値になる
- 提示のたびに、要点2〜3個と「次に決めるべきこと」を短く添える。
  文書を投げて終わりにしない
- ユーザーの決定は必ず文書に反映してから次工程へ進む
