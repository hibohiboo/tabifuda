# 横断的関心事文書の作り方

## なぜ準備段階で必要か

権限・ログ・削除方針は「後で足せる」ように見えて、しばしば
ドメインモデルに逆流する(例: 認可はルールの一部なのでコアで検証
すべき→Command/Eventにアクターが必要)。準備段階で一度検討し、
逆輸入を済ませてから実装に入る方が安い。

## 検討項目チェックリスト

1. **権限(認可)**
   - 認証(誰か)と認可(何ができるか)を分離。認可のロジックの置き場を決める
   - コアで検証するなら: Actor/Roleの型、コマンドごとの権限規則表、
     Forbidden系エラー、テストは受理/拒否を対で
2. **ログの2分類**
   - 製品ログ(履歴・監査・リプレイ)と運用ログ(障害調査)は別物
   - 運用ログの禁止事項を先に決める: ユーザー本文・個人情報を書かない。
     IDと種別のみ。本文が要る調査はIDで製品ログを引く
   - この規律は最初のCLI/ローカル段階から習慣化する
3. **自由入力・UGC**
   - すべての自由入力箇所を列挙する(見落としがちな箇所: 作者名、
     メモ、コメント、タイトル)
   - 描画時エスケープ既定、生HTML挿入の禁止を明文化
   - 「ユーザーデータはコードではない」: 実行機構(eval、スクリプト、
     テンプレート展開)を持たないならそれを設計特性として記録する
   - 長さ上限を型レベルで
4. **削除要求 vs 不変履歴**
   - 追記のみ・イベントソーシング・監査ログ系の設計は必ずこの緊張を持つ
   - 精密化の型: 「構造は不変、本文は削除・伏字可能」
     (Redactイベント+物理上書きのハイブリッド等)
   - マルチユーザーなら「自分の削除が他人の記録を壊さない」を要件にする
5. **秘密情報**: .env(gitignore)→本番はシークレットマネージャ。
   シークレットスキャンをCIへ(P0級に早くてよい)
6. **依存監査**: audit系をCI警告→どの時点で必須化するか
7. **フェーズ対応表**で締める(どの項目がいつ効くか)

## やらないことの明示

準備段階では方針だけ決め、実装(認証基盤、WAF、レート制限等)は
該当フェーズに送る。ただし「方針未定のまま実装が進む」ことだけは防ぐ。
