# テスト戦略の作り方

## 文書の構成

1. 方針を一行で(重複防止の原則を先頭に宣言)
2. リスク表(リスク / 深刻度 / テストする層 / 手法)
3. レイヤ別の書き方
4. 重複を作らないためのルール
5. フェーズ別の導入順
6. CIゲート
7. エージェント運用との接続

## 大原則

**ロジックの置き場所にテストの質量を置く。**
コアに全ロジックを寄せた設計なら、テストの8割はコア単体に置き、
外側の層(CLI/UI/API)は「翻訳」のテストだけ書く。
ルールの正しさを外側の層で再テストしたら、それは重複であり
「外側にロジックが漏れている」設計上の警告でもある。

## リスク表の作り方

- 深刻度「高」= 起きたらユーザーデータや製品価値が壊れるもの
  (例: 保存データが読めなくなる、履歴の非決定性、再開不能)
- 深刻度が高いリスクほど、安く速い層(単体・純粋関数)で
  テストできる位置に設計を寄せられないか先に検討する
- UIの見た目など深刻度「低」は自動化を後回しにしてよいと明記する

## 手法の使い分け

| 手法 | 使いどころ |
|---|---|
| テーブル駆動 | 操作ごとの受理/拒否。**必ず対で書く**規約にする |
| プロパティテスト | 不変条件の一般則(決定性、保存則、状態機械の合法性、往復同値) |
| ゴールデン/リプレイ | 代表シナリオの通し記録をfixture化。回帰+直列化互換+スモークの3役 |
| 契約テスト | API層。「コアの結果を正しく運ぶか」「認可」「エラー変換」のみ |
| 統合テスト(実物) | モック不可の箇所限定(同時実行制御、トランザクション挙動) |
| E2E/スモーク | フェーズごとに1本まで。通しの動作確認のみ |

## モックしてはいけない箇所

検証したい性質がその依存先の挙動そのものである場合、モックすると
テストが無意味になる。典型例: DBの楽観ロック/一意制約/分離レベル、
ファイルシステムの同時書き込み。ここだけは実物(コンテナ等)で回す。
数を絞れば常時CIで許容できる。

## 重複防止ルール(文書に必ず書く)

1. ルール分岐のテストは中核層のみ
2. 境界値はテーブル駆動、一般則はプロパティ、と役割分担
   (同じ条件を両方で細かく書かない)
3. fixtureは1系統を多目的に使い回す
4. 仕様変更PRは設計文書とテストの変更を必ず両方含む

## エージェント運用との接続

- 機械的な規約(受理/拒否を対で書く等)は安価モデルでもレビュー可能に
  なるので、規約として明文化する価値が高い
- 失敗ジャーナル由来の「テストがあれば防げた」項目を
  プロパティテストへ昇格する経路を書いておく
