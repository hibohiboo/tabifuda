# エージェント運用計画の作り方

## 必ず最初にやること

モデルのラインナップ・価格・期間限定価格をWeb検索で確認する。
記憶で価格表を書かない(価格・モデルは数ヶ月で変わる)。
確認した日付と出典URLを文書に残す。

## 3層モデルラダー

| 層 | 役割 | 選定基準 |
|---|---|---|
| 主力(既定) | 設計・実装・レビューの大半 | 価格性能比が最良の現行モデル |
| 定型用 | lint修正、体裁、検証実行、下調べ | 最安の実用モデル |
| エスカレーション用 | 難所のみスポット | 主力の1〜2段上。既定にしない |

最上位ティアのモデルは、コスト要件がある限り原則使わない。

### 定型用モデルに回すタスクの目安
コミットメッセージ、CHANGELOG、fmt/lint修正、データ検証の実行と修正、
テスト雛形、文書の体裁統一、サブエージェントでのファイル探索。

### エスカレーション発動条件(必ず明文化する)
1. 主力モデルが同一タスクに2回失敗(2ストライク・ルール)
2. 誤ると波及の大きい設計判断のレビュー(型設計、DBスキーマ、
   公開API、IaCセキュリティ)
3. フェーズ移行時のアーキテクチャレビュー(各フェーズ1回まで)

## フェーズ×モデル対応表

プロジェクトのフェーズ計画に沿って表を作る。原則:
**上流(ドメインモデル・型設計)ほど上位モデルの価値が高く、
下流(実装・設定)ほど安いモデルで十分。**
上流の誤りは全フェーズに波及するため、そこだけはケチらない。

## 開発サイクル

1サイクル=1つの縦切り機能。順序を固定する:
要件追記 → 設計文書更新(実装より先にコミット)→ 実装(TDD推奨)→
検証(テスト+文書乖離チェック)→ ふりかえり

「文書が正、実装が従」を守ると、エージェントに渡すコンテキストが
常に信頼できる状態に保たれる。これが安いモデルで回せる前提条件。

## コンテキスト管理

- 一次対策はタスク粒度: 1サイクルはコンテキスト60%以内に収まる大きさで切る
- 保険として60%ハンドオフ: WIPコミット→使い捨てメモ(目標/完了済み/
  残作業/次の一手/ハマりどころ/パス)→新セッション→引き継ぎ後メモ削除
- 規律: メモに仕様を書かない(正の二重化禁止。仕様は設計文書へ)
- 自動圧縮の発動はタスク設計の失敗としてジャーナルに記録

## ハーネス改良ループ

- 失敗ジャーナル: エージェントの誤解・失敗を1行/件で記録
  (何を指示し、何を誤ったか)。記録自体は定型用モデルでよい
- 週次棚卸し: 各失敗を「CLAUDE.md修正/スキル新設/タスク分割変更」の
  どれで防げたか分類し、ハーネス変更をPRとして出す(コードと同じレビュー)
- 「テストがあれば防げた誤り」はテスト(可能ならプロパティテスト)に昇格

## コスト運用

- CLAUDE.mdは安定内容に保つ(プロンプトキャッシュのため)
- 急がない一括処理はBatch API(50%引)を検討
- 月次で使用量を確認。エスカレーション用モデルの比率に上限
  (目安2割)を置き、超えたら発動条件を見直す
- 期間限定価格・トークナイザ変更等の「期限つき前提」はTODOとして
  日付つきで文書に残す
