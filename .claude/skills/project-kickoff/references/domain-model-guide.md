# ドメインモデル叩き台の作り方

## 文書の構成(この順で)

1. 冒頭: 対象範囲(どのコンポーネントか、どのMVPまでか)と変更履歴
2. 基本原則(3〜5個。アーキテクチャの背骨)
3. 中核概念ごとの節(型スケッチつき)
4. 状態と遷移(状態機械があるなら遷移図をテキストで)
5. 操作の一覧(コマンド/イベント、またはAPI相当)
6. MVPでの簡略化(何を落とすか明示)
7. 未決事項(相談したい)— 必須

## イベントソーシングを提案する条件

以下のどれかが要件にあれば、decide/apply型の設計を第一候補にする:
- 履歴・ログ・リプレイ・ふりかえりが製品価値に直結する
- 「途中で止めて後で再開」「非同期に手番が進む」がある
- 監査・由来追跡(誰がいつ何をしたか)が要る

その場合の骨格:
```
decide(state, command) -> Result<Vec<Event>, Error>   // 検証と意思決定
apply(state, event) -> State                          // 状態遷移(必ず成功)
```
原則: 乱数・時刻はイベントに焼き込む(決定性)。状態の直接変更禁止。

該当しないなら無理に採用しない。CRUD+履歴テーブルで足りる場合は
その旨を理由つきで書く。

## 型スケッチの書き方

- 主要な構造体/enumを5〜10個、コードブロックで示す(コンパイル可能で
  なくてよい。コメントで意図を書く)
- ID類はnewtypeを推奨
- enumは将来の追加が予想されるものに印をつけ、シリアライズ方針
  (タグ付き表現、non_exhaustive相当)を注記する
- 「スナップショット(凍結コピー)か参照か」を、共有される
  データすべてについて明記する。並行実行・後からの編集と独立させたい
  ものはコピーと書く

## 未決事項の書き方

- 「決めの問題」を3個前後に絞る(多すぎると議論が発散する)
- 各項目: 選択肢、推奨案、推奨理由、後から変えるコスト(小/中/大)
- 工数を大きく左右する項目には目印をつけ、ユーザーの判断を促す

## 改訂(v0.1以降)の作法

- 冒頭に「vXからの変更点」を箇条書きで残す
- 他文書(将来要望メモ等)由来の変更は出典を明記する
- ユーザーが「別文書に退避」と言った要望を本文に混ぜない
