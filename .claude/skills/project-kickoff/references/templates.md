# 文書テンプレート

## future-requirements.md(将来要望メモ)

```markdown
# 将来要望メモ vN

現行の設計には含めないが、設計判断時に壊さないよう意識しておく要望集。
各項目に「現行モデルへの影響」を付記する。

## N. <要望名>

要望: <ユーザーの言葉に近い形で>

設計方針(案): <将来どう実現するかの1案。現行の仕組みの延長で
説明できるならその対応関係を書く>

現行モデルへの影響: なし/小/中/大。<根拠。「〜と明記すれば現行構造の
まま成立」「〜の変更は早めが楽」等>

派生論点(未決): <あれば>

## 現行版への逆輸入候補(小さい予約)

実装コストほぼゼロで将来の道を塞がない予約を抽出する。典型:
1. 空フィールドの先行追加(直列化互換の管理が楽になる)
2. 「参照ではなくコピー」等の意味論の明記
3. enum/種別の追加前提シリアライズ方針の明文化
```

要望の聞き取り時、「この段階で考慮しなくてよい」とユーザーが言った
内容を現行文書に混ぜない。逆輸入候補だけは反映可否を確認する。

## CLAUDE.md の骨格

```markdown
# CLAUDE.md

<プロジェクト一行説明>

## 最重要ルール
<最大4つ。文書が正・アーキテクチャ原則・迷ったら質問、など>

## リポジトリ構成
<ツリー+各ディレクトリ一行説明。未着手はフェーズ注記>

## 必読文書(タスク種別ごと)
<「〜に触れるなら docs/... を読む」の対応表>

## コマンド
<テスト・lint・fmt。コミット前必須のものを明示>

## <言語>規約
<enum拡張方針、ID型、エラー方針、テストの対規約など5個前後>

## 用語(揺らさない)
<用語 | 意味 の表。混同しやすい対を優先>

## やらないこと
<禁止事項。将来要望の先回り実装禁止を必ず含める>

## 作業の終わり方
<テスト実行、文書乖離チェック、失敗ジャーナルへの自己申告>
```

原則: 全体で100行前後に収める。揮発する情報(進捗、期限、価格)は
入れない。それらは docs/ へ。

## docs/ ディレクトリの標準配置

docs/
