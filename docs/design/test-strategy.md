# テスト戦略 v0

方針を一行で: **ロジックはコアにしか置かない。だからテストの質量もコアに置く。**
外側の層(CLI/WASM/API/UI)は「翻訳」しかしないので、翻訳のテストだけを書く。
ルールの正しさを外側の層で再テストしない——これが重複防止の唯一にして最大の原則。

## リスク分析とテストの置き場所

| リスク | 深刻度 | テストする層 | 手法 |
|---|---|---|---|
| ルールの誤り(decide/apply) | 高 | core単体 | テーブル駆動+プロパティ |
| リプレイ非決定性(ログ=冒険記が壊れる) | 高 | core単体 | プロパティ+ゴールデン |
| パッチ適用でセッションが壊れる(再開不能) | 高 | core単体 | プロパティ |
| シリアライズ互換の破壊(過去セッションが読めない) | 高 | core単体 | スナップショット+旧データ読込 |
| シナリオデータの不備(到達不能シーン等) | 中 | データ検証 | シナリオlint |
| WASM境界の型変換ミス | 中 | engine-wasm | 往復テスト少数 |
| API層(認可・同時追記の競合) | 中 | api統合 | 契約テスト+DB統合 |
| インフラ設定ミス | 中 | デプロイ後 | スモーク最小限 |
| UIの表示崩れ | 低 | 手動+スモーク | 自動化は後回し |

## レイヤ別の書き方

### 1. crates/core(テストの8割をここに)

**(a) テーブル駆動の decide テスト**(CLAUDE.md規約の具体化)
- 各Commandに「受理→期待イベント列」と「拒否→期待RuleError」を必ず対で書く
- 拒否系の網羅対象: Paused中のPlayCard/Propose、条件未達カード、
  存在しないシーンへのGotoScene、Ended後の全Command

**(b) プロパティテスト**(proptest)。守るべき不変条件:
1. **決定性**: 同じイベント列を2回applyすれば同じ状態(fold等価性)
2. **整合性**: decideが返すイベント列は、必ずエラーなくapplyできる
3. **状態機械の合法性**: どんなコマンド列でも Running/Paused/Ended の
   遷移図に無い遷移は起きない
4. **保存則**: カードは配布イベント無しに手札に現れない/消えない
5. **パッチ安全性**: validateを通ったパッチは、適用後も
   「現在シーンが存在」「配布済みカードの定義が解決可能」を壊さない
- ランダム生成するのはコマンド列とパッチ。シナリオは小さな固定物+生成物の両方
- Vec/HashMapを複数階層ネストする型に `Arbitrary` をderiveする場合、
  各コレクションフィールドに最初からサイズ上限を付ける
  (例: `#[proptest(strategy = "proptest::collection::vec(..., 0..=3)")]`)。
  無制限だと生成物が指数的に膨張し、テスト1回が数十分・数GBになりうる
  (2026-07-18に実例。上限設定で約26分→約1秒)

**(c) ゴールデン(リプレイ)テスト**
- テンプレシナリオ「単純討伐」の通しプレイをイベント列JSONとして
  fixtures/ に保存。リプレイ結果の最終状態をスナップショット比較
- 兼用効果: リグレッション検知+シリアライズ後方互換の実地検証。
  **enumに種別を追加したらこのfixtureが読めることがそのまま互換性テストになる**
- 配置(P2 C4決定): `crates/tabifuda-core/fixtures/`(イベント列本体と、
  最終`Session`状態のスナップショットの2ファイル)。テストは`include_str!`
  でコンパイル時に埋め込む(coreに実行時ファイルIOを持ち込まないため)。
  fixtureは意図的に凍結する対象なので、シナリオや型を変えても自動
  再生成はしない(壊れたら「何が非互換になったか」を人間が判断してから
  更新する)

**(d) シリアライズ往復**
- 全公開型に serialize→deserialize→同値 のroundtripテスト(プロパティで一括)

### 2. シナリオデータ(lintをテストとして実行)

同梱テンプレシナリオ全件に対し:
- 全カード/シーン/フラグ参照が解決できる
- オープニングから到達可能なシーンのみで構成(到達不能=警告)
- どのシーンからも Ended に到達する経路がある(詰み検知)
- 将来 `scenario-validate` スキルの中身とこのlintは同一実装を共有する

### 3. engine-cli / engine-wasm(薄く)

- CLI: 標準入出力の結合テストを1〜2本(通しプレイのスモーク)。
  ルール分岐はテストしない(coreで済んでいる)
- WASM: TS↔Rust境界の型往復テストのみ(Command/Eventが正しくJSON越えするか)。
  wasm-bindgen-test で数本

### 4. apps/api(P4〜)

- **契約テスト**: エンドポイントごとにHTTPレベルの入出力を検証。
  ルールの正誤ではなく「coreの結果を正しく運ぶか」「認可」「エラー変換」を見る
- **イベントストア統合テスト**: 楽観ロック(セッションのversion+追記の競合)を
  実DBで検証。ここだけはモックにしない(競合はDBの挙動そのもの)。
  ローカルPostgres(コンテナ or PGlite)で実行
- サーバ側でもcoreのdecideを通す設計なので、不正コマンド拒否のテストは
  「decideが呼ばれてエラーがHTTPに変換される」1系統だけでよい

### 5. E2E/スモーク(最小限)

- P2: CLIの通しプレイ(上記(c)の生成元を兼ねる)
- P3: Webでテンプレシナリオを1本通すPlaywrightスモーク1本
- P5: デプロイ後に「セッション開始→カード1枚→ログ取得」のcurlスモーク

## 重複を作らないためのルール

1. ルール分岐のテストは core のみ。外側の層に if 分岐が生えたら設計の警告
2. 同じ不変条件を単体とプロパティの両方で細かく書かない。
   境界値はテーブル駆動、一般則はプロパティ、と役割分担する
3. fixtureは1系統(テンプレシナリオのイベント列)を多目的に使い回す
   (リグレッション/互換性/CLIスモークの3役)
4. 「テストが仕様」としてエージェントに扱わせる: 仕様変更PRは
   設計文書とテストの変更を必ず含む。テストだけ直して文書を放置しない(逆も)

## CIゲート

- 常時: fmt / clippy(-D warnings)/ cargo test --workspace / シナリオlint
- P4以降: DB統合テストはサービスコンテナで常時実行(数が少ないので許容)
- 任意(余裕があれば): cargo-mutants をP1完了時に一度回し、
  テストの穴を測る。常時実行はしない

## エージェント運用との接続

- テストはエージェントに対する最強のガードレール。
  「decideの受理/拒否を対で書く」等の機械的規約はHaikuでもレビュー可能
- 失敗ジャーナルで「テストがあれば防げた誤り」が出たら、
  その不変条件をプロパティテストに昇格させる(ハーネス改良と同じ週次で)
