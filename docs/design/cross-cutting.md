# 横断的関心事 v0(ログ・セキュリティ・削除方針)

対象: 全レイヤ。個別機能ではなく、全フェーズの実装が従う方針集。

## 権限

- 権限検証はdecide内で行う(ドメインモデルv0.2参照)。API層は
  「認証(誰か)」だけを担い、「認可(何ができるか)」はcoreに委ねる
- サーバ側(P4〜)でも必ずcoreのdecideを通す。クライアントの検証は
  UX目的の先行チェックにすぎず、信頼しない

## 2種類のログを区別する

| | ドメインログ(冒険記) | 運用ログ |
|---|---|---|
| 目的 | 製品価値。リプレイ・ふりかえり | 障害調査・監視 |
| 置き場 | イベントストア(Neon) | CloudWatch(P5〜) |
| 内容 | Event列そのもの | 構造化JSON(リクエストID、セッションID、コマンド種別、結果、所要時間) |
| 禁止事項 | — | **自由入力テキスト・カード本文を書かない**(個人情報混入経路を断つ) |

- 運用ログにはIDと種別だけを書く。本文が必要な調査はIDでドメインログを引く
- P2(CLI)の時点から、コアの呼び出し結果を種別+IDで記録する薄い
  ログ層を挟み、この規律を先に習慣化する

## 自由入力(UGC)の取り扱い

対象: 台詞カードのfree_text、提案text、パッチのnote、
シナリオ作者のカード名・narration等すべて。

1. **描画時エスケープを既定とする**。ReactのデフォルトでXSSは防げるが、
   `dangerouslySetInnerHTML` と生HTML挿入を禁止事項として明文化する。
   装飾が欲しくなったら生HTMLではなく限定マークアップ(サブセット)を検討
2. **シナリオはコードではない**。Effect/Conditionの閉じたenumのみで、
   スクリプト実行機構を持たない。これは意図したセキュリティ特性であり、
   「シナリオに自由なスクリプトを」という将来要望が来たら
   この文書を根拠にトレードオフを議論する
3. 長さ上限を型レベルで設ける(free_text等)。DoSと保存コストの上限。
   機構は `BoundedString<const MAX: usize>` newtype(custom Deserialize で
   crate 境界越しに境界を強制。`try_new` は Result で panic なし。用途別上限を
   型に持たせる)。段階適用(2026-07-18決定。レビューL2):
   - **実行時の自由入力**(`PlayCard.free_text`=C2、`Propose.text`=C3、
     `ScenarioPatch.note`=C4)から先に適用する(DoS面が最も切実)
   - **作者データ**(カード名・text・narration・title・author)は P2
     (テンプレシナリオ+シナリオlint登場時)に段階適用する
   - 上限値(chars/bytes の基準含む)は導入サイクルで確定する
4. 公開コンテンツ(シナリオ、公開冒険記)の通報・非公開化は将来要望
   メモ側に追記(P4以降の運用機能)

## 削除要求と追記のみ原則の折り合い

イベントログは追記のみだが、自由入力には個人情報が入りうる。方針:

- **打ち消しではなく「編集イベント」**: `TextRedacted { event_ref, by }` を
  追加し、リプレイ時は対象イベントのfree_textを伏字で表示する。
  元本文はイベントストア上で**物理上書き**する(参照整合は保ったまま
  本文だけ消す)。ログの構造・決定性は保たれ、本文だけが消える
- アカウント削除時: そのユーザーのfree_text等を一括Redact+
  ユーザー↔キャラクターの対応を仮名化。冒険記の構造は残る
  (他の参加者の記録を壊さないため)
- この設計により「追記のみ」の対象は**構造**であり、**本文は削除可能**、
  と原則を精密化する。domain-model側の「過去改変禁止」はこの精密化を含む

## 秘密情報・設定

- APIキー等はコードとdocsに書かない。ローカルは.env(gitignore)、
  P5ではSSM Parameter Store/Lambda環境変数
- CIにシークレットスキャン(gitleaks等)を早期に入れる(P0で設定可能)

## 依存関係

- cargo audit / pnpm audit をCIの警告ゲートに(P0で設定、P4から必須化)
