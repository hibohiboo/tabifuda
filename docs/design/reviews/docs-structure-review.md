# docs/ 構造レビュー(進化型設計との適合性)

日付: 2026-07-19 / レビュー対象: docs/ 配下の全文書(26ファイル)と
その構造・粒度。実装コードは対象外。
観点: (1) 進化型設計(設計が実装とともに進化する前提の文書運用)に
合っているか (2) 不足はないか (3) 内容の粒度がおかしい点はないか。

**各項目の「対処方針」は提案であって決定ではない。** 対応要否・優先度は
人間が判断する(採用する場合は decision-log 運用に乗せることを推奨)。

---

## 総評

構造は進化型設計に**強く適合している**。変化を取り込む配管——ADR(決定の
経緯)、決定ログ(判断の証跡)、失敗ジャーナル(傾向分析)、ふりかえり→
反映先の表、design-sync(乖離チェック)、README の読者別地図と更新規律——が
一通り揃い、実際に運用された実績もある(例: ADR 0005 で過去の不正確な根拠を
訂正、phase2 ふりかえりの気づきが反映先へ展開済み)。「正を二重化しない」
原則が繰り返し明文化され、YAGNI(future-requirements.md での先回り実装禁止)
も守られている。

主な問題は「不足」よりも**「現在の仕様」と「決定の経緯」の混在**
(domain-model.md の肥大化と時系列軸の節構成)と、**フェーズ全体像の
集約先が無い**ことの2点。以下、重要度順。

---

## 🔴 High

### H1. フェーズ全体計画(ロードマップ)を1枚で見られる文書が無い ✅ 対応済み(2026-07-19)

**指摘**: P0〜P5 の目的・完了条件・順序は tasks/phaseN-task.md ×6 に
分散しており、「なぜこの6フェーズか」「いまどこか」を一覧する文書が無い。
さらにフェーズ対応情報が cross-cutting.md「フェーズ対応」表・
test-strategy.md「フェーズ別の導入順」表・agent-operations.md
「フェーズ×モデル対応」表の3箇所に分散している。

**進化型設計上の問題**: 計画は変わる前提だからこそ、変わったときに直す
集約先が要る。分散表の更新漏れは既に実害が出ている(不変条件5のP1前倒しで
test-strategy.md の表が古いまま残った件。agent-journal.md 2026-07-19、
P1 C5 の記録)。

**対応**: docs/roadmap.md を作成(非規範の索引。各フェーズの正は
tasks/phaseN-task.md と明記し、正を二重化しない)。各フェーズの目的・
完了条件の1行要約+状態列と、分散する3つのフェーズ対応表への参照一覧を
置いた。表そのものの統合はしない(各表は各文書の文脈で意味を持つため、
「前提が変わったらここから辿って点検する」というインデックスだけ集約)。
README の進め方視点の表にも1行追加した。

### H2. domain-model.md が「現在の仕様」と「決定の経緯」の混在で肥大化 ✅ 対応済み(2026-07-19)

**指摘**: 620行(次点の2.8倍)。以下の3点で規範文書としての引きやすさが
落ちている。

1. **節がサイクル番号軸**: 「C2: decide/applyの解決規則」「C3: …」
   「C4: …」という節名はタスク履歴の軸であり、仕様を主題(コマンド・概念)
   で引けない。例: `Propose` の仕様は「セッション状態機械」
   「コマンドとイベント」「C3節」の3箇所に分散。C2/C3 が何かは
   tasks/phase1-task.md を読まないと分からない
2. **経緯句の混入**: 「2026-07-19決定」「ユーザーへ確認の上」「レビューH1」
   等の経緯が本文に多数埋め込まれている。README の優先順位ルールは
   「adr/ は決定の経緯、design/ は正」と分けているのに、正の側に経緯が
   積もる非対称がある
3. **冒頭の版履歴**: 「v0.1からの変更点」「v0からの変更点」リストが残る。
   phase0-task.md は「版は git 履歴で管理に切替」と定めており不整合

**進化型設計上の問題**: 規範文書は「最新仕様のスナップショット」であるべきで、
履歴は git・ADR・決定ログが担う——という自らの分担方針からの乖離。
このまま P3〜P5 の決定が積もると、読む・直すコストが線形以上に増える。

**対応**: 一括で再構成した(仕様内容は不変。構成のみ変更)。
- C2/C3/C4節を「進行の解決規則」(StartSession / シーン入場と
  CardInstanceId / PlayCard / カードの消費・除去 / 提案と裁定 /
  GmAdvance / 共通の拒否系)と「validate の解決規則」「ApplyPatch の実行」
  (シナリオパッチ配下)へ主題別に再編
- 経緯句(日付・レビュー番号・確認の経緯)は本文から除き、末尾
  「決定の経緯(参照)」の表に集約。過去文書(決定ログ・ジャーナル・ADR)は
  凍結記録として更新せず、「旧節名との対応」表で参照を吸収
- 冒頭の版履歴(v0.1/v0からの変更点)を削除(git 履歴へ。正規化済みの
  内容は本文の各節が保持)
- シナリオlint仕様を design/scenario-lint.md へ分離(規範文書に昇格。
  README の規範文書リスト・開発者視点表に追加)
- 生きている参照は同時更新: コード内コメント5ファイル(engine.rs /
  session.rs / primitives.rs / lint.rs / lint_tests.rs)と
  design-sync-screen エージェント。domain-guide.md 用語対応表が参照する
  節名(「カード」「シナリオ構造」「セッション状態」「セッション状態機械」
  「コマンドとイベント」「Target の意味論」「シナリオパッチ(構造化)」
  「アクターと権限」「カードの消費・除去」)はすべて維持

## 🟡 Medium

### M1. requirements/ に「現在の要件」の置き場が無い ✅ 対応済み(2026-07-19)

**指摘**: requirements/ は future-requirements.md(将来要望)のみ。
実装済みになった要望はメモから消える運用(旧§6 カード除去)のため、
「いま何を作ることになっているか」の要件レベルの記述は domain-model.md
「ソロMVPでの簡略化」節と domain-guide.md に散在する。
agent-operations.md「開発サイクルの回し方」手順1は「要件: docs/requirements/
に追記」と定めるが、追記先ファイルが実質未定義。

**対応**: (b) を採用(現在の規模では専用ファイルは過剰)。宣言を3箇所で
整合させた: future-requirements.md 冒頭に位置づけ(将来要望専用。現在の
要件の正は design/ 規範文書と tasks/。採用された要望は規範文書へ移して
メモから削除、実例: 旧§6)を明記 / README の同文書の行に補足 /
agent-operations.md「開発サイクルの回し方」手順1の追記先を
future-requirements.md と明示し、着手済み要望の移動規律を追加。

### M2. ADR 化の基準が未定義で、決定記録の粒度が不均一 ✅ 対応済み(2026-07-19)

**指摘**: ADR 0005(ProposalId発番=実装レベルの決定)がある一方、
同格以上の決定——flags 廃止(「世界はすべてカード」への統一)、Target の
意味論、role 信頼モデル、カード消費ルール——は ADR が無く、
domain-model.md 内の節+決定ログ/レビュー記録にのみ経緯がある。
「なぜこうなっているかを遡る」入口として adr/ 一覧が機能しない
(README は adr/ をその用途として案内している)。

**対応**: adr/0001-methodology.md に「ADR化の基準」節を追加(基準2件:
複数フェーズ/レイヤへの波及、過去の根拠を覆す訂正。該当しない決定は
決定ログ+規範文書の反映で足りると明記し、過去決定の遡及ADR化はしないと
明言)。agent-operations.md「人間の判断が要る論点の進め方」に手順6として
「ADR化の要否を確認する」を追加し、決定ログのクローズ時に基準へ照らす
運用にした。flags廃止・Target意味論・role信頼モデル・カード消費ルール等の
既存決定は遡及ADR化しない(基準の明文化自体が目的で、過去の粒度不均一は
許容する)。

### M3. agent-operations.md の見出し欠落(構造バグ) ✅ 対応済み(2026-07-19)

**指摘**: 「開発サイクルの回し方」手順5が「ふりかえり(ハーネス改良):
下記」と参照するが、該当内容(失敗ジャーナル・週次棚卸し・スキル・
補助エージェントの箇条書き。現在の141〜169行付近)に見出しが無く、
「コンテキスト管理とハンドオフ」節の末尾にぶら下がって見える。
文書内リンク・目次生成でも拾えない。

**対応**: 箇条書き群の直前に「## ふりかえり(ハーネス改良)」見出しを補った
(手順5の参照表記「ふりかえり(ハーネス改良): 下記」と同一の名称に揃えた)。

### L1. フロント層(CLI/Web)の設計文書の置き場が未定義

**指摘**: domain-model.md「カードの消費・除去」節に「CLIの手札表示からは
Marker を除外する」という CLI 表示決定が書かれている(本文自身が
「規範ではなく CLI の表示ロジックの決定」と断っている=置き場が無いことの
兆候)。P3 で wasm 境界・UI 設計が始まると同種の決定が増える。

**対処方針(案)**: 今は移動不要。P3 C1 の設計時に design/ 配下の
レイヤ別文書(例: wasm-boundary.md、frontend.md)を作るかを判断し、
作ったら Marker 表示の記述も移す。phase3-task.md への1行追記で予約できる。

### L2. 非規範文書(domain-guide.md / demo.md)の追従が仕組み化されていない

**指摘**: 非規範文書の乖離は「見つけたら報告」のみで、design-sync の
対象外。実例: カードの消費・除去(P2後に導入)が domain-guide.md の
遊び方説明に未反映(矛盾ではなく欠落——使用カードが消えること、Marker が
手札表示に出ないことに触れていない)。

**対処方針(案)**: phase-cycle スキル「終わり方」またはふりかえり手順に
「domain-guide.md / demo.md への影響確認」を1行足す。非規範文書に厳密同期を
求めない(多少の欠落は許容する)という選択も可——その場合はこの指摘を
「対応不要」でクローズしてよい。

### L3. docs/handoff/ が README の地図に載っていない

**指摘**: CLAUDE.md と agent-operations.md が参照する docs/handoff/ は
使い捨て運用のため通常存在しない(現在も無い)。README の一覧にも無いため、
初見の読者・エージェントが「参照先が無い」と混乱しうる。

**対処方針(案)**: README の進め方視点の表に「handoff/ — ハンドオフ用の
一時メモ置き場(通常は空。運用は agent-operations.md)」の1行を足す。

## 適合している点(変更不要。維持すべき性質)

- **変化の配管が完備**: ADR / 決定ログ / ジャーナル / ふりかえり→反映先 /
  design-sync。進化型設計の核である「変わったときに追従する仕組み」が
  文書運用として実装され、稼働実績がある
- **規範/非規範の明示**: 各文書冒頭の位置づけ宣言と README の優先順位
  ルールにより、矛盾時の解決手順が機械的に決まる
- **README の更新規律**(文書追加時に地図へ1行)と読者別入口
- **YAGNI の徹底**: future-requirements.md が「作らないが壊さない」を
  現行モデルへの影響付きで管理し、逆輸入は小さい予約に限定
- **決定の訂正が一級市民**: 不正確だった根拠を ADR 0005 として訂正・記録
  した運用は、進化型設計の「決定は暫定であり更新される」をそのまま体現
