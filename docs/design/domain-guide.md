## 1. Tabifuda は何のソフトか

カード制TRPG(CardWirth風)。**シナリオ作者**が書いた冒険の台本を、
**GM(進行役)** と **プレイヤー** が一緒に遊ぶ。ソロMVPでは1人が
GMとプレイヤーを兼任する。

3つの登場物を区別すると全体が見える:

| 登場物 | 例えるなら | 特徴 |
|---|---|---|
| **シナリオ** | 演劇の台本 | 作者が事前に書くデータ。誰が演じるかは知らない |
| **セッション** | 一回の上演 | 台本+実際の出演者(パーティ)で行う1回のプレイ |
| **冒険記** | 上演の録画 | セッション中の出来事の記録。あとから再生(リプレイ)できる |

重要な性質: セッション開始時に台本とパーティを**凍結コピー**する。上演中に
原作が改訂されても、進行中の上演には影響しない。

## 2. 世界はすべてカード

プレイヤーが行うことは、原則「手札からカードを1枚出す」だけ。
カードの種類で表現の幅を出す:

| カード種別 | 何を表すか | 例 |
|---|---|---|
| 行動(Action) | キャラ固有の技能 | 「剣を振る」「鍵開け」 |
| シナリオ(Scenario) | 台本が配る選択肢 | 「依頼を受ける」「勝利」「敗北」 |
| 台詞(Dialogue) | 発言。自由文を添えられる | 「よろしく頼む」+一言 |
| 提案(Proposal) | 台本にない行動の提案 | 「屋根から侵入したい」→GMが裁定 |
| アイテム(Item) | 所持品 | 「回復薬」「洞窟の鍵」 |
| 目印(Marker) | 世界の状態・選択の記録 | 「街の門が開いた」「Aルートを選んだ」 |

「フラグ管理」のような見えない内部変数は持たない方針。世界の状態も
Markerカードとして卓上に見える形で置く(2026-07-18決定)。

出したカードは手札に残らない。台詞・シナリオカードは使ったその場で消え、
選ばなかった側の選択肢カード(例: クライマックスで「敗北」を選ばなかった
場合の「敗北」カード)もシーンを離れた時点で自動的に片付く。行動
(Action)・アイテム・目印(Marker)は消えない。ただし目印カードは「選んだ
記録」であり、プレイヤーが選ぶ対象ではないため手札の一覧には表示されない
(実装の詳細は domain-model.md「カードの消費・除去」)。

## 3. 1プレイの流れ(テンプレシナリオ「単純討伐」)

```
オープニング: 酒場のシーン
  → シナリオが「依頼を受ける」カードを配る
  → プレイヤーが出す → 効果「次のシーンへ」

ミドル: 街道のシーン(移動・準備)

クライマックス: 戦闘のシーン
  → シナリオが「勝利」「敗北」の2枚を配る
  → プレイヤーがどちらかを選んで出す → 効果「対応する結末シーンへ」

エピローグ: 勝敗に応じた結末 → セッション終了
```

進行の裏側はすべて「出来事(イベント)」として記録される。
「カードが配られた」「カードが出された」「シーンが変わった」…の列が冒険記になり、
同じ列を再生すれば必ず同じ結果になる(決定性)。

## 4. カードの「効果」と「対象」— Target という概念

カードを出すと**効果(Effect)**が起きる。効果には種類がある:

- シーンを移動する(GotoScene)
- 幕を進める(AdvancePhase)
- **カードを配る(DealCard)** ← 誰に?
- **能力値を増減する(ModifyStat)** ← 誰の?
- セッションを終える(EndSession)

このうち「カードを配る」「能力値を増減する」は**相手が要る**効果で、
その相手の指定方法が **Target(対象)** という概念。

### 対象の指し方は2種類(2026-07-18決定)

台本(シナリオ)は上演前に書かれる。つまり作者は執筆時点で
**出演者(キャラクター)の名前を知らない**。そこで指し方を区別する:

| 指し方 | 例えるなら | 使える場所 |
|---|---|---|
| **役割参照** `Party`(パーティ全員) | 台本に「一同に配る」と書く | 台本の中でOK |
| **実名参照** `Character(id)` | 「田中さんに渡す」 | **上演中だけ**(GMの改編・その場の配布)。台本に書いたら不正(検査で弾く) |

テンプレシナリオ「単純討伐」の配布(依頼・勝利・敗北カード)はすべて
`Party` で書ける。「カードを出した本人に渡す」のような他の役割参照は、
必要になった時点で追加する(追加しても既存データは壊れない設計)。
決定の経緯: tasks/plans/p1-c1-review-decisions.md の #1。

## 5. GMの介入: 提案・休止・パッチ

- プレイヤーが**提案**カードを出すと、セッションは**休止(Paused)**する
- GMは提案を**採用/却下**で裁定する。採用にあたって台本を書き直したければ、
  **パッチ**(構造化された差分。シーン追加・カード追加など)を当ててから再開する
- 却下されても提案の記録は冒険記に残る(「提案したが見送られた」も冒険の一部)

この仕組みで「台本にない展開」を安全に取り込める。パッチは適用前に検証され、
「現在いるシーンを消してしまう」ような壊れた改編は拒否される。

## 6. 権限: 誰が何をできるか

| 操作 | プレイヤー | GM |
|---|---|---|
| カードを出す/提案する | ○(担当キャラのみ) | ○ |
| 提案の裁定/パッチ適用/強制進行 | × | ○ |

権限チェックはUIではなく**ルールエンジンの中**で行う(画面側をすり抜けても
不正操作はできない)。役割は上演の参加者名簿(roles)だけが決め、操作のたびに
「GMです」と名乗って通る仕組みは存在しない(2026-07-18決定)。
GMはプレイヤーの操作(カードを出す等)も行えるため、ソロMVPでは
1人をGMとして登録するだけでよい。
