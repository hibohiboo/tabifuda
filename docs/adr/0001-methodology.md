# ADR 0001: 開発手法の明文化

状態: 採用 / 日付: 2026-07-12

## 文脈

本プロジェクトは単一の方法論ではなく複数の手法を組み合わせている。
エージェント・人間の双方が「いま何に従っているか」を参照できるよう明文化する。

## 決定

以下のブレンドを採用する。

1. **Functional Core, Imperative Shell**(構造の背骨)
   - crates/core は純粋。IO・時刻・乱数を持たない
   - 外側(CLI/WASM/API/UI)は翻訳のみ。テスト戦略はこの配置に依拠
2. **イベントソーシング+軽量CQRS**(状態管理)
   - decide/apply。全進行はイベント。リプレイ決定性を不変条件とする
3. **DDD 戦術パターンの部分採用**(モデリングの語彙)
   - ユビキタス言語(CLAUDE.mdの用語表)、Sessionを集約=整合性境界、
     ドメインイベント
   - 戦略的設計(境界づけられたコンテキスト分割)は**未実施**。
     P4で「シナリオ編集/セッション進行/公開・コミュニティ」の
     境界を検討する(ADR候補)
4. **ドキュメント駆動**: 設計文書が正、実装が従。仕様変更は文書先行
5. **リスクベーステスト**+**縦切りイテレーション**(1サイクル=1機能)
6. **エージェント運用**: モデルラダー+失敗ジャーナル駆動のハーネス改良
   (docs/agent-operations.md)

## ADR化の基準(2026-07-19追加)

決定の記録先が「domain-model.md『決定の経緯』表+決定ログ」で足りるか、
ADR を起こすべきかが暗黙のままだった(実例: ADR 0005 ProposalId発番は
ADR化された一方、同格以上の flags 廃止・Target の意味論・role 信頼モデル・
カード消費ルールは domain-model.md 内の記述+決定ログのみで ADR が無い。
docs構造レビュー M2)。以下のいずれかに当てはまる決定は ADR を起こす:

1. **複数フェーズ/複数レイヤに波及する決定**(例: パッケージマネージャ、
   CIパイプライン、ハーネス設定。単一クレート・単一節の変更で閉じない)
2. **一度採った根拠を覆した・訂正した決定**(例: ADR 0005。「なぜ今の形か」
   を後から遡るとき、訂正の経緯自体が価値を持つ)

当てはまらない決定(coreの型・状態機械・解決規則などdomain-model.mdの
記述範囲に閉じるもの)は、決定ログ(agent-operations.md「人間の判断が
要る論点の進め方」)で証跡を残し、規範文書の該当節+「決定の経緯」表
(domain-model.mdの例)へ反映すれば足りる。ADRにはしない。

**過去の決定の遡及ADR化はしない。** 経緯は決定ログ・reviews/・
agent-journal.md に残っており、追加コストに見合わない。

## 帰結

- 「フルDDDか」と問われたら否。戦術のみ部分採用と答える
- コアに副作用を持ち込む変更、外側の層にルール分岐を書く変更は
  本ADRへの違反として却下する
- 戦略的設計の要否はP4入り口で再評価する
- 決定ログの論点がクローズする際、上記基準に該当するなら決定ログとは別に
  ADRを起こす(agent-operations.md「人間の判断が要る論点の進め方」参照)
