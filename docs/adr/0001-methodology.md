# ADR 0001: 開発手法の明文化

状態: 採用 / 日付: 2026-07-12

## 文脈

本プロジェクトは単一の方法論ではなく複数の手法を組み合わせている。
エージェント・人間の双方が「いま何に従っているか」を参照できるよう明文化する。

## 決定

以下のブレンドを採用する。

1. **Functional Core, Imperative Shell**(構造の背骨)
   - crates/core は純粋。IO・時刻・乱数を持たない
   - 外側(CLI/WASM/API/UI)は翻訳のみ。テスト戦略はこの配置に依拠
2. **イベントソーシング+軽量CQRS**(状態管理)
   - decide/apply。全進行はイベント。リプレイ決定性を不変条件とする
3. **DDD 戦術パターンの部分採用**(モデリングの語彙)
   - ユビキタス言語(CLAUDE.mdの用語表)、Sessionを集約=整合性境界、
     ドメインイベント
   - 戦略的設計(境界づけられたコンテキスト分割)は**未実施**。
     P4で「シナリオ編集/セッション進行/公開・コミュニティ」の
     境界を検討する(ADR候補)
4. **ドキュメント駆動**: 設計文書が正、実装が従。仕様変更は文書先行
5. **リスクベーステスト**+**縦切りイテレーション**(1サイクル=1機能)
6. **エージェント運用**: モデルラダー+失敗ジャーナル駆動のハーネス改良
   (docs/agent-operations.md)

## 帰結

- 「フルDDDか」と問われたら否。戦術のみ部分採用と答える
- コアに副作用を持ち込む変更、外側の層にルール分岐を書く変更は
  本ADRへの違反として却下する
- 戦略的設計の要否はP4入り口で再評価する
