## 1. セッションの中断と並行プレイ

要望:
- プレイヤーはセッションを中断できる
- 中断中、別パーティで別セッション、あるいは同じパーティを別セッションで使える
- ただし他セッションで得たカード等は、再開時に反映されない

設計方針(案): **キャラクターもスナップショット方式にする**
- シナリオと同様、セッション開始時にパーティ/キャラクターを凍結コピーして
  Session内に保持する(v0.1のScenarioSnapshotと同じ考え方)
- マスターデータのキャラクターとセッション内キャラクターを分離。
  セッション中の成長・取得はセッション内コピーにのみ反映
- 報酬(§3)はセッション終了処理でのみマスターへ書き戻す

現行モデルへの影響: 小。`Session.party` を「参照」ではなく「コピー」と明記すれば
v0.1の構造のまま成立する。イベントソーシングとも整合(セッションは自己完結)。

派生論点(未決):
- 同一パーティの並行参加を許すと、報酬の書き戻しが競合しうる。
  「同一キャラは同時に1セッションまで」等の制約を入れるか、
  終了順に単純マージするか。

## 2. キャンペーンシナリオ/セッション

要望: 前のセッションを攻略したパーティのみ参加できるセッション。

設計方針(案):
- シナリオmetaに `prerequisites: Vec<Requirement>` を追加
  (例: `ClearedScenario(ScenarioId)`)
- パーティ/キャラクターのマスターデータに攻略履歴
  `cleared: Vec<(ScenarioId, Outcome, SessionId)>` を持たせる
- セッション開催時(StartSession)の検証で判定

現行モデルへの影響: なし(マスターデータ層とシナリオmetaの拡張のみ)。
攻略履歴は§3のセッション終了処理で記録するのが自然。

## 3. 報酬とセッション終了処理

要望:
- クリア報酬としてカードの持ち出し、キャラメイク用ポイントの付与
- 持ち出せないものはセッション終了時に破棄する仕組み

設計方針(案):
- CardDefに `portable: bool`(または後述タグで `#portable`)を持たせる
- シナリオに報酬定義: `rewards: Vec<Reward>` 
  (`GrantCard(CardId)`, `GrantPoints(u32)` など。Outcome別に定義できるとよい)
- セッション終了処理(finalize)を明示的な段階として設ける:
  1. Outcome確定
  2. 報酬確定(シナリオ定義+持ち出し可カードの選別)
  3. マスターデータへ書き戻し(キャラ成長・攻略履歴・カード付与)
  4. 持ち出し不可カードの破棄
- finalizeもイベントとして記録する(`RewardsGranted`, `CardsDiscarded`)

現行モデルへの影響: 小。`SessionEnded` の後段に finalize フェーズを足すだけ。
§1のスナップショット方式が前提(書き戻しポイントが1箇所に定まる)。

## 4. カードのタグシステム

要望: シナリオがカードのタグを見て処理できる。
例: 蜘蛛の巣で塞がれた道は、`#炎` タグを持つ任意のカードで焼き払える。

設計方針(案):
- `CardDef.tags: Vec<Tag>` を追加(Tagは文字列ベースの新型)
- Conditionに `HasCardWithTag(Tag)` を追加
- シーン側に「タグ受け」の仕組み: シーンやオブジェクトカードが
  `accepts: [{ tag: 炎, effects: [...] }]` を持ち、該当タグのカードを
  出すとそのeffectsが発動する(カード自身のeffectsとは別系統)
- 汎用性の核: カード作者は用途を知らずにタグを付け、シナリオ作者は
  カードを知らずにタグで受ける。疎結合になる

現行モデルへの影響: **v0.1の時点で `tags` フィールドだけ予約しておく価値あり**
(空でよい)。シリアライズ形式に後から足すより、最初からある方が
シナリオデータの互換性管理が楽。

## 5. 条件付き選択肢とテキスト変化

要望:
- 特定カード所持、シナリオ経験、クリアフラグで選択肢が増える
- キャラクター属性(性別など)でNPCの会話内容が変わる
- シーン分岐まではいかない「会話が少し変わるだけ」も表現したい

設計方針(案):
- 選択肢の条件: v0.1の `CardDef.requires: Vec<Condition>` がそのまま土台。
  Conditionを拡張する:
  - `ClearedScenario(ScenarioId)` (§2と共通)
  - `CharacterAttr(AttrId, AttrValue)` (性別等。Characterに属性mapを追加)
  - 「条件を満たさないカードを 非表示にする/グレーアウトで見せる」は
    カード側で選べるとよい(見えるが選べない、は誘い文句として有効)
- テキスト変化: narrationや台詞を条件付きフラグメントの列にする
  ```
  narration: [
    { text: "店主が顔を上げた。" },
    { text: "「おや、また来たのかい」", if: ClearedScenario(prev) },
    { text: "「いらっしゃい、初めての顔だね」", unless: ClearedScenario(prev) },
  ]
  ```
  分岐(別シーン)を作るほどでもない差分を安価に表現できる

現行モデルへの影響: 中。narrationを `String` から
`Vec<ConditionalText>` に変える判断は早めが楽(v1入り口で検討)。
Conditionのenum拡張は互換性を壊さない。

## 6. フォーク公開とシナリオ作者への還流

要望:
- GMがセッション中に改編(パッチ)したシナリオを、次のセッションでも使いたい
- シナリオ作者は、GMたちの改編を差分としてレビューし、元シナリオに取り込みたい

実装済みの範囲(v0.2、CLIフォーク出力):
- セッション終了後、パッチ適用済みスナップショットを別シナリオとして保存できる
  (`meta.forked_from`で由来を追跡。domain-model.md「フォーク出力」参照)

残る要望(本節の対象):
- **パッチ列形式の永続化**: 「元シナリオ+パッチ列」をそのまま保存・公開する
  フォーマット。スナップショットと違い、元シナリオの更新への追従や
  差分レビューが可能になる
- **作者への還流ワークフロー**: フォークの差分(パッチ列)を作者が
  レビューして本家へ取り込む流れ(Pull Request相当)。P4のサーバ側
  (共有・配布)とセットで検討
- 複数セッションのパッチの合成・衝突解決

現行モデルへの影響: 小。`ScenarioMeta.forked_from`と構造化パッチ
(`ScenarioPatch`)が既に土台。パッチ列の永続フォーマットと、
`validate`を「セッション状態なし(シナリオ単体)」でも使える形に
整理する程度。
