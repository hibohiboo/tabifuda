## #1 ProposalId発番方式(C3再決定・ADR 0005化)

**論点**: P1 C3で決定した「`Session.proposal_seq: u64` による連番発番
(decideが `proposal-{seq}` を発行、applyの `ProposalSubmitted` 処理で+1)」は
妥当か。phase2完了を機にユーザーから再検討の提起があった:

1. core内でカウンタをインクリメントして発番するのは煩雑ではないか
2. 外部でUUIDを発行してCommandに添付しても純粋性には影響しないのでは
3. 発番ロジック自体をcoreから返す(呼び出し側が取得して添付)形もありうる

検討結果はADRとして記録する(ADR 0005予定)。

**現行実装**(2026-07-19時点):

- `Session.proposal_seq: u64`(session.rs)。`decide_propose` が
  `ProposalId(format!("proposal-{}", session.proposal_seq))` を発行して
  `ProposalSubmitted{id, ..}` に焼き込み(engine.rs:283)、applyが
  `proposal_seq += 1`(engine.rs:131)
- `CardInstanceId` も同型の設計(`Session.card_instance_seq`、
  `{CardId}-{連番}` 形式。domain-model.md C2)
- `pending_proposal` は `JudgeProposal` 裁定後に `None` へ戻るため、
  **過去に発行したProposalIdは現在状態に残らない**(これが「現在状態からの
  逆算」を不可能にし、カウンタ導入の動機となった。C3決定の経緯)

**選択肢**:

- (A) **現行維持**: `Session.proposal_seq` による連番発番を維持。
  ただしdomain-model.md C3の「UUIDは純粋性に抵触するため不採用」という
  根拠記述は不正確(下記論点整理1)なので、ADRで正しい根拠に差し替える
- (B) **外部発行ID**: `Command::Propose` にProposalIdフィールドを追加し、
  外部(CLI/API層)がUUID等を発行して添付。`proposal_seq` は削除
- (C) **core提供の発番関数**: `next_proposal_id(&Session) -> ProposalId` の
  ような純粋関数をcoreが公開し、呼び出し側が取得してCommandに添付。
  カウンタは維持

**論点整理**(2026-07-19、エージェント提示):

1. **純粋性の観点では(B)(C)とも問題ない。** 外部発行のUUIDはただのデータ
   として渡り、core内で乱数を引くわけではない。IDはイベントに焼き込まれる
   のでリプレイ決定性も保たれる。→ 現行C3の「UUID(v4等)はcoreの純粋性
   (乱数禁止)に抵触するため不採用」は「**core内で**生成する場合」に限った
   話であり、記述として不正確。**どの案を採っても訂正が必要**
2. **一意性の保証責務**: 過去のProposalIdは状態に残らない(上記)ため、
   (B)ではcoreが重複IDを検出できない(検出には「発行済みID全集合」を
   状態に持つ必要があり、u64カウンタより重い)。バグったクライアントが
   同一IDを2回送ってもcoreは受理し、ProposalIdをキーに冒険記を読む側
   (リプレイUI等)が壊れる。(A)なら一意性はcoreの不変条件として閉じる
3. **CardInstanceIdへの一般化可能性**: `DealCard` のTarget解決
   (Party→人数分)やシーン入場配布では、**必要なID個数がdecide内部の
   状態解決まで分からない**(engine.rs:246-249等)ため、外部からID列を
   事前に渡すことは原理的に不可能。(B)を採るとProposalIdだけ外部発行と
   なり発番戦略が二本立てになる
4. **(C)の評価**: カウンタは結局Sessionに残る。呼び出し側が正しく
   呼ぶことを信頼するか、coreが照合検証するかの二択で、検証するなら
   添付の意味がなく往復が1回増えるだけ。(A)の劣化版になる
5. **煩雑さの実態**: フィールド1個(`u64`)+applyの `+= 1` 1行。
   「ProposalSubmittedイベントの累計数」というイベント列のfoldとして
   自然な導出状態
6. **クライアント発行IDの本来の利点**(リトライの冪等化)は、coreが
   重複検出できない以上core層では実現できず、P4のAPI層の冪等キーとして
   扱うのが適所

**論点整理への追補**(2026-07-19、ユーザー提起「UUID v4/v7の衝突可能性は
無視できるほど小さいのでは」への応答):

7. **偶発的衝突の確率は論点2の核心ではない — 提起を認めて再定式化する。**
   正しく生成されたUUID v4/v7の偶発的衝突(v4で約2^-122)は無視してよく、
   「衝突するかもしれないから危険」という枠組みならユーザー指摘のとおり
   成立しない。論点2の核心は確率ではなく**検証可能性**:
   - coreは受け取ったIDが「新規に正しく生成されたUUID」か「過去IDの再利用・
     固定文字列」かを**区別できない**(型は `ProposalId(String)`。UUID形式の
     検査は可能だが、正規のUUIDの再利用は形式検査では防げない)。
     一意性が「coreが維持する不変条件」から「全呼び出し側の行儀についての
     仮定」に変わる
   - 想定すべき失敗モードは偶発的衝突ではなく、リトライ実装のバグ
     (同一Commandオブジェクトの再送)、コマンド列の複製、P4(マルチ
     プレイヤー/API公開)での悪意ある入力
   - これは **H2決定(Actor.role自己申告の廃止)と同型**。本プロジェクトは
     「冗長データを常に検証で守るのではなく、不正状態を型・構造で表現
     不能にする」を既に選択している(p1-c1-review-decisions.md #4)。
     呼び出し側申告のIDはrole自己申告と同じ「coreが検証できない自己申告」
   - **譲歩**: 「ID一意性はホスト層の責務」と割り切る設計はevent-sourced
     システムとして一般的であり、案Bでもシステム自体は成立する。争点は
     確率ではなく設計方針(coreがどの不変条件を所有するか)である
   - なお論点3(CardInstanceIdへの一般化不能)はUUIDを使っても解消しない
     (必要個数がdecide内部でしか決まらないのはID形式と無関係)

**エージェント推奨**: (A)。根拠は純粋性ではなく上記2(一意性の責務)と
3(一般化不能)。ADRでは「不採用理由の訂正」を明記する。
