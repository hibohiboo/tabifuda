# Phase 2 実装タスク: コンソール版ソロプレイMVP

実行モデル: Sonnet 5。1サイクル=1セッション=1PR。
**開始前の儀式(全フェーズ共通)**: CLAUDE.md と docs/design/ の関連文書を読む。
本タスクは事前に書かれたものなので、P1完了時点の実装と食い違う記述が
あれば、作業前に人間へ差分を報告し、本タスク文書を改訂してから着手する。

## 目的
テンプレシナリオ「単純討伐」を tabifuda-cli で通しプレイできる状態にする。
Phase 1 のコアに手を入れる場合は設計文書更新を先行させる。

## サイクル

### C1: シナリオデータ形式とローダ+シナリオlint
- シナリオのファイル形式(JSON)を確定し、scenarios/ ディレクトリに配置する規約を決める
- lint実装(test-strategy.md §2): 参照解決 / 到達可能性 / 詰み検知。
  テストとして実行+`tabifuda-cli lint <file>` サブコマンド
- lintは将来スキル(scenario-validate)と実装共有する前提で、コアまたは
  専用モジュールに置く(CLI固有にしない)

### C2: テンプレシナリオ「単純討伐」
- OP(依頼会話・台詞カード使用)→ミドル(移動)→クライマックス(勝利/敗北
  カード選択)→エピローグ分岐→終了。提案→GM裁定が最低1回試せる構成
- lint全通過。データレビューを人間に依頼して確定

### C3: CLIプレイループ
- シーン描写・手札表示・カード選択・自由入力・提案/裁定(ソロ=両ロール)
- 運用ログ規律の開始(cross-cutting.md): 種別+IDのみを記録する薄いログ層。
  自由入力本文をログに書かないことをテストで固定
- 対話部はコアから分離(翻訳のみ。ルール分岐を書いたら設計違反)

### C4: 冒険記出力とゴールデンfixture
- セッション終了後、イベント列から冒険記(テキスト)を描画
- 通しプレイのイベント列JSONを fixtures/ に保存し、リプレイ最終状態の
  スナップショットテストを追加(回帰・直列化互換・スモークの3役)
- CLIスモークテスト(標準入出力の結合テスト)1〜2本

## 完了条件
単純討伐を最初から最後まで遊べる / lint・全テスト通過 /
fixtureが存在しCIで検証されている
