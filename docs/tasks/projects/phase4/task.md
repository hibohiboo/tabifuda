# Phase 4 実装タスク: バックエンド(Hono+Drizzle+Neon)

実行モデル: Sonnet 5。1サイクル=1セッション=1PR。
開始前の儀式は phase2/task.md 冒頭と同じ。
**フェーズ入り口で2つの上流判断を行う(Opus 4.8レビュー対象)**:
1. DBスキーマ+イベントストア設計
2. 境界づけられたコンテキストの要否(ADR 0001の宿題:
   シナリオ編集/セッション進行/公開・コミュニティの分割)

## 人間の事前決定(ADR化必須)
- 認証方式(推奨候補: メールマジックリンク or OAuth。実装コストと
  運用負荷を比較提示して人間が選ぶ)
- サーバ側でのルール検証方法(推奨: tabifuda-wasm を Node/Lambda 上で
  実行し、クライアントと同一のdecideを通す。採用前に起動時間・実行時間を
  簡易ベンチして確認)

## サイクル

### C1: packages/schema(Drizzle)
- users / characters / parties / scenarios / scenario_forks(パッチ列) /
  sessions(version付き) / events(追記のみ、session_id+seq一意)
- マイグレーション生成、ローカルPostgres(コンテナ)で適用テスト

### C2: イベントストアとセッションAPI
- コマンド受付API: 認証→Actor構築→(wasm)decide→events追記。
  **楽観ロック(sessionのversion)をDB実物で統合テスト**(モック禁止)
- セッション作成/参加/状態取得/イベント取得(冒険記)

### C3: シナリオCRUDとフォーク
- 公開/下書き、フォーク=元ID+パッチ列、シナリオ→セッション一覧
- lint をサーバ側でも実行(投稿時検証)

### C4: 非同期マルチの手番と提案フロー
- Paused/裁定/再開をAPI越しに。通知は当面ポーリング(将来要望に
  プッシュ通知を追記)
- 契約テスト(認可・エラー変換。ルールの再テスト禁止)

### C5: 削除・Redactフロー
- cross-cutting.md §削除 の実装: TextRedactedイベント+本文物理上書き、
  アカウント削除の一括Redact+仮名化。統合テスト必須

## 完了条件
Web版がAPI経由で動く / 2ユーザーの非同期セッションが成立 /
楽観ロック・削除フローが実DBテストで固定されている
