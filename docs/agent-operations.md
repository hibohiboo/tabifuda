# エージェント運用計画 v0

目的: 「AIエージェントのスキル・ハーネス・エージェント自体を改良しながら、
要件定義→設計→実装のサイクルを回し続ける」ための運用ルールとモデル配分。

前提: Claude Code を主ハーネスとする。Fable 5($10/$50 per MTok)は使わない。

## タスク種別ごとの割り当て

### Sonnet 5(既定)
- 設計文書の起草・更新、実装、テスト作成、通常のコードレビュー
- リファクタリング、WASMバインディング、Drizzleスキーマ実装

### Haiku 4.5(安く数を回す)
- lint/fmt修正、コミットメッセージ、CHANGELOG
- シナリオデータ(JSON/RON)のバリデーション実行と修正
- 「設計文書と実装の乖離チェック」の一次スクリーニング
- テストの雛形生成、ドキュメントの体裁統一
- サブエージェントとしての探索(ファイル調査、grep的な下調べ)

### Opus 4.8(エスカレーション専用)
発動条件を決めておく(気分で使わない):
1. Sonnet 5が同じタスクに2回失敗した(2ストライク・ルール)
2. 型設計・不変条件など、誤ると後で高くつく設計判断のレビュー
   (例: Event/Command設計、パッチのvalidate仕様、DBスキーマ確定、IaCのセキュリティ)
3. フェーズ移行時のアーキテクチャレビュー(各フェーズ1回まで)

## フェーズ×モデル対応

| フェーズ | 主力 | Opusスポット投入箇所 | Haiku担当 |
|---|---|---|---|
| P0 骨格・ハーネス | Sonnet 5 | なし | CI設定の微修正 |
| P1 コアのドメインモデル | Sonnet 5 | **Event/State/Patch型設計レビュー(最重要)** | テスト雛形 |
| P2 CLI版MVP | Sonnet 5 | なし | シナリオデータ作成・検証 |
| P3 WASM+Web UI | Sonnet 5 | wasm境界API設計レビュー | UI文言・体裁 |
| P3.5 CLI永続化 | Sonnet 5 | finalize(報酬・持ち帰り)のEvent設計レビュー | セーブ/パーティのフィクスチャ作成 |
| P4 Hono+Drizzle+Neon | Sonnet 5 | DBスキーマ+イベントストア設計レビュー | マイグレーション検証 |
| P5 AWSデプロイ | Sonnet 5 | IaCセキュリティレビュー | 設定ファイル整備 |

原則: **上流(P1)ほどOpusの価値が高く、下流ほどSonnet/Haikuで十分**。
型とイベント設計の誤りは全フェーズに波及するため、P1のレビューだけはケチらない。

## 開発サイクルの回し方

1サイクル = 1つの縦切り機能(例:「Proposalで休止→パッチ→再開」)とする。

1. **要件**: 要望を docs/requirements/future-requirements.md に記録
   (人間+Sonnet 5)。requirements/ は**将来要望専用**であり、
   着手が決まった要望は次ステップで規範文書へ移し、メモから削除する
   (「現在の要件」の正は design/ の規範文書と tasks/ が兼ねる)
2. **設計**: plan modeで設計案を出させ、docs/design/ を先に更新。
   実装より設計文書のコミットが先(文書が正、実装が従)
3. **実装**: Sonnet 5。TDD推奨(コアは純粋関数なのでテストが書きやすい)
4. **検証**: `cargo test` + Haikuによる「設計文書との乖離チェック」
5. **ふりかえり(ハーネス改良)**: 下記

- 長時間になりうるコマンド(テスト全体、生成系処理など)は、経過が逐次見える
  形で実行するか、先に小規模ケースで所要時間を見積もってから流す。
  バックグラウンドで最後に一括出力する構成は、暴走(実行時間・メモリの膨張)の
  検知が遅れる(2026-07-18のproptest 26分暴走が実例)

## フェーズ完了時のふりかえり(2026-07-19手順化)

フェーズの**最終サイクルが完了したら、人間の指示を待たずに**ふりかえりを
作成する(phase-cycleスキル「終わり方」に組み込み済み)。P2で指示待ちに
なった反省から手順化した。

1. **ドラフト生成**: `retrospective` エージェント(.claude/agents/)に
   フェーズ名を渡してドラフトを作らせてよい(git log・agent-journal.md・
   タスク文書から機械的に集める部分の省力化)。セッション内の文脈で
   書ける場合は直接書いてもよい
2. **docs/retrospectives/phaseN.md に保存**。構成は固定:
   成果(完了条件の充足) / うまくいったこと / 課題(ジャーナル参照) /
   気づきと対応(反映先の表)
3. **「気づきと対応」は必ず反映先の文書を決めて同PRで反映する**
   (将来機能→future-requirements.md、次フェーズの進め方→次のphaseN-task.md、
   運用改善→本文書またはスキル)。ふりかえり文書は経緯の記録であり、
   対応内容の正は反映先に置く(正を二重化しない)
4. **docs/roadmap.md の「状態」列を更新する**(フェーズ全体像の索引を
   現在地に追従させる)
5. 人間に報告し、フェーズ移行の判断(次フェーズ着手・アーキテクチャ
   レビュー要否)を仰ぐ

## 人間の判断が要る論点の進め方(決定ログ)

対象: レビュー指摘・設計上の選択など、**人間の判断待ちがセッションを跨ぎうる
論点**。チャットのやり取りは揮発するため、進捗と決定の証跡は必ずgitに残す。

1. **決定ログを作る**: `docs/tasks/plans/<対象>-decisions.md` に論点を列挙する。
   各論点は「論点 / 選択肢 / 決定(未定) / 反映先」を自己完結で書き、
   新しいセッションがその文書だけで再開できる状態にする
2. **状態は4段階**: 未着手 → 議論中 → 決定済み → 反映済み。冒頭に状態表を置く
3. **状態が変わるたびにコミット**: 「議論中」で中断しても、どこまで話したかが
   git履歴で追えるようにする。決定時は決定内容と理由も文書に記録する
4. **1論点ずつ進める**: 「①論点提示 → ②人間が決定 → ③文書・実装へ反映 →
   ④決定ログ更新」を論点単位で区切り、まとめて聞かない
5. **正を二重化しない**(ハンドオフと同じ規律): 決定した仕様は設計文書
   (docs/design/ 等)へ反映するのが正。決定ログは経緯と証跡であり、
   仕様の置き場ではない。全論点が反映済みになったらログはクローズ
   (削除せず残す。後から「なぜこうなったか」を引くため)
6. **ADR化の要否を確認する**: 論点が adr/0001-methodology.md「ADR化の基準」
   (複数フェーズ/レイヤへの波及、または過去の根拠を覆す訂正)に該当するなら、
   反映と同時にADRも起こす。該当しなければ決定ログ+規範文書の反映で足りる
   (該当しない決定を遡ってADR化する必要は無い)

初出の実例: docs/tasks/plans/p1-c1-review-decisions.md(P1 C1型設計レビュー)。

## コンテキスト管理とハンドオフ

自動圧縮は「何を残すか」を制御できない不可逆要約であり、圧縮後の
エージェントは知らないことを知らない。以下の二段構えで防ぐ。

**一次対策: タスク粒度。**
1サイクルはコンテキスト60%以内に収まる粒度で切る。60%に頻繁に
達するなら、それは引き継ぎの出番ではなくタスクの切り方が大きすぎる
シグナル(分割を先に検討)。

**二次対策(保険): 60%ハンドオフ。**
使用率60%に達したら作業を止め、次を行って新セッションへ移る:
1. WIPコミットを切る(新セッションが git diff とテスト結果から
   状態復元できるようにする。文書への依存を減らす)
2. docs/handoff/ に使い捨てメモを書く。項目は固定:
   目標 / 完了済み / 残作業 / 次の一手 / ハマりどころ / 関連ファイルパス
3. 新セッションはメモ+CLAUDE.md+該当設計文書から再開し、
   引き継ぎ完了後にメモを削除する

**規律: 正を二重化しない。**
ハンドオフメモに仕様・決定事項を書かない。書きたくなったら、それは
設計文書(docs/design/ 等)に属する内容なので先にそちらへ移す。
ハンドオフは経緯のメモであり、仕様の置き場ではない。

**計測: 自動圧縮に到達したら失敗。**
圧縮が発動したケースは agent-journal.md に記録し、週次棚卸しで
タスク設計の失敗として扱う。

## ふりかえり(ハーネス改良)

- **失敗ジャーナル**: エージェントが誤解・失敗したら docs/agent-journal.md に
  1行で記録(何を指示し、何を誤ったか)。Haikuに記録させてよい
- **週次で棚卸し**: ジャーナルを見て、CLAUDE.md修正/スキル新設/
  タスク分割の変更のどれで防げたかを分類し、ハーネスをPRとして更新する。
  ハーネス変更もコードと同じくレビュー対象
- **スキル(`.claude/skills/`、git管理)**: 実装済み(2026-07-13)は
  1. `phase-cycle`: docs/tasks/ のフェーズタスクを1サイクル実行する手順
     (開始の儀式・スコープ規律・停止ポイント・終わり方)
  2. `design-sync`: 設計文書と実装の乖離チェック手順
  3. `core-invariants`: decide/applyの不変条件一覧(テスト観点)
  4. `handoff`: 60%ハンドオフの実行手順(渡す側・受ける側)
  5. `decision-log`(2026-07-18): 人間の判断が要る論点の決定ログ運用
     (上記「人間の判断が要る論点の進め方」が正)
- 残候補: `scenario-validate` はP2のシナリオlintと同一実装を共有する前提のため、
  P2 C1(lint実装)完了後に作成する
- **スキルには手順と参照先のみを書き、仕様を二重化しない**(正は docs/)。
  docs側を変えたら該当スキルも同PRで更新する
- スキルの評価にはskill-creatorの評価手順を利用できる
- **補助エージェント(`.claude/agents/`、git管理。2026-07-19導入)**:
  定型の下請け作業をサブエージェントとして切り出したもの(ADR 0004参照)。
  サブエージェントは実行中にユーザーへ対話的に質問できないため、
  **人間の判断を要する作業(サイクル実行そのもの)はエージェント化しない**
  (2026-07-19決定。メインセッション+phase-cycleスキルで行う)。実装済み:
  1. `design-sync-screen`(Haiku・読取専用): 設計文書と実装の乖離チェックの
     一次スクリーニング。疑わしい箇所の列挙までを担い、乖離の判定と修正は
     メインセッションが design-sync スキルで行う
  2. `retrospective`(Sonnet・読取専用): フェーズふりかえりのドラフト生成。
     保存・反映先への反映・報告はメインセッションが行う
     (上記「フェーズ完了時のふりかえり」参照)

## コスト運用ルール

- プロンプトキャッシュ前提でCLAUDE.mdは安定した内容に保つ(頻繁に書き換えない。
  揮発する情報はdocs/へ)
- コンテキストに巨大ファイルを丸ごと入れない。docs/は目的別に分割
- 急がない一括処理(シナリオ量産、ログ整形など)が発生したらBatch API(50%引き)を検討
- 月初にAPIコンソールで使用量を確認し、Opus比率が2割を超えていたら
  エスカレーション条件を見直す
