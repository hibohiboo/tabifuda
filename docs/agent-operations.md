# エージェント運用計画 v0

目的: 「AIエージェントのスキル・ハーネス・エージェント自体を改良しながら、
要件定義→設計→実装のサイクルを回し続ける」ための運用ルールとモデル配分。

前提: Claude Code を主ハーネスとする。Fable 5($10/$50 per MTok)は使わない。

## タスク種別ごとの割り当て

### Sonnet 5(既定)
- 設計文書の起草・更新、実装、テスト作成、通常のコードレビュー
- リファクタリング、WASMバインディング、Drizzleスキーマ実装

### Haiku 4.5(安く数を回す)
- lint/fmt修正、コミットメッセージ、CHANGELOG
- シナリオデータ(JSON/RON)のバリデーション実行と修正
- 「設計文書と実装の乖離チェック」の一次スクリーニング
- テストの雛形生成、ドキュメントの体裁統一
- サブエージェントとしての探索(ファイル調査、grep的な下調べ)

### Opus 4.8(エスカレーション専用)
発動条件を決めておく(気分で使わない):
1. Sonnet 5が同じタスクに2回失敗した(2ストライク・ルール)
2. 型設計・不変条件など、誤ると後で高くつく設計判断のレビュー
   (例: Event/Command設計、パッチのvalidate仕様、DBスキーマ確定、IaCのセキュリティ)
3. フェーズ移行時のアーキテクチャレビュー(各フェーズ1回まで)

## フェーズ×モデル対応

| フェーズ | 主力 | Opusスポット投入箇所 | Haiku担当 |
|---|---|---|---|
| P0 骨格・ハーネス | Sonnet 5 | なし | CI設定の微修正 |
| P1 コアのドメインモデル | Sonnet 5 | **Event/State/Patch型設計レビュー(最重要)** | テスト雛形 |
| P2 CLI版MVP | Sonnet 5 | なし | シナリオデータ作成・検証 |
| P3 WASM+Web UI | Sonnet 5 | wasm境界API設計レビュー | UI文言・体裁 |
| P4 Hono+Drizzle+Neon | Sonnet 5 | DBスキーマ+イベントストア設計レビュー | マイグレーション検証 |
| P5 AWSデプロイ | Sonnet 5 | IaCセキュリティレビュー | 設定ファイル整備 |

原則: **上流(P1)ほどOpusの価値が高く、下流ほどSonnet/Haikuで十分**。
型とイベント設計の誤りは全フェーズに波及するため、P1のレビューだけはケチらない。

## 開発サイクルの回し方

1サイクル = 1つの縦切り機能(例:「Proposalで休止→パッチ→再開」)とする。

1. **要件**: docs/requirements/ に追記(人間+Sonnet 5)
2. **設計**: plan modeで設計案を出させ、docs/design/ を先に更新。
   実装より設計文書のコミットが先(文書が正、実装が従)
3. **実装**: Sonnet 5。TDD推奨(コアは純粋関数なのでテストが書きやすい)
4. **検証**: `cargo test` + Haikuによる「設計文書との乖離チェック」
5. **ふりかえり(ハーネス改良)**: 下記

## コンテキスト管理とハンドオフ

自動圧縮は「何を残すか」を制御できない不可逆要約であり、圧縮後の
エージェントは知らないことを知らない。以下の二段構えで防ぐ。

**一次対策: タスク粒度。**
1サイクルはコンテキスト60%以内に収まる粒度で切る。60%に頻繁に
達するなら、それは引き継ぎの出番ではなくタスクの切り方が大きすぎる
シグナル(分割を先に検討)。

**二次対策(保険): 60%ハンドオフ。**
使用率60%に達したら作業を止め、次を行って新セッションへ移る:
1. WIPコミットを切る(新セッションが git diff とテスト結果から
   状態復元できるようにする。文書への依存を減らす)
2. docs/handoff/ に使い捨てメモを書く。項目は固定:
   目標 / 完了済み / 残作業 / 次の一手 / ハマりどころ / 関連ファイルパス
3. 新セッションはメモ+CLAUDE.md+該当設計文書から再開し、
   引き継ぎ完了後にメモを削除する

**規律: 正を二重化しない。**
ハンドオフメモに仕様・決定事項を書かない。書きたくなったら、それは
設計文書(docs/design/ 等)に属する内容なので先にそちらへ移す。
ハンドオフは経緯のメモであり、仕様の置き場ではない。

**計測: 自動圧縮に到達したら失敗。**
圧縮が発動したケースは agent-journal.md に記録し、週次棚卸しで
タスク設計の失敗として扱う。



- **失敗ジャーナル**: エージェントが誤解・失敗したら docs/agent-journal.md に
  1行で記録(何を指示し、何を誤ったか)。Haikuに記録させてよい
- **週次で棚卸し**: ジャーナルを見て、CLAUDE.md修正/スキル新設/
  タスク分割の変更のどれで防げたかを分類し、ハーネスをPRとして更新する。
  ハーネス変更もコードと同じくレビュー対象
- **スキルの初期候補**:
  1. `design-sync`: 設計文書と実装の乖離チェック手順
  2. `scenario-validate`: シナリオデータのスキーマ検証と代表エラーの直し方
  3. `core-invariants`: decide/applyの不変条件一覧(テスト観点)
- スキルの評価にはskill-creatorの評価手順を利用できる

## コスト運用ルール

- プロンプトキャッシュ前提でCLAUDE.mdは安定した内容に保つ(頻繁に書き換えない。
  揮発する情報はdocs/へ)
- コンテキストに巨大ファイルを丸ごと入れない。docs/は目的別に分割
- 急がない一括処理(シナリオ量産、ログ整形など)が発生したらBatch API(50%引き)を検討
- 月初にAPIコンソールで使用量を確認し、Opus比率が2割を超えていたら
  エスカレーション条件を見直す
